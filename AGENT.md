# AGENT.md — エージェント向けリポジトリガイド

このファイルは、AI エージェントがこのリポジトリで作業する際のルールと規約を定めています。

---

## 基本ルール

- **コミット前にユーザーの許可を得る。** ファイルの作成・編集が完了したら「コミットしてよいですか？」とユーザーに確認を取る。許可を得るまで `git add` / `git commit` / `git push` は実行しない。
- **コンテンツはすべて日本語で作成する。** コードコメント、説明文、見出し、エラーメッセージの解説を含む。ただし、コード識別子（変数名・クラス名など）は英語のまま使用する。
- 不明な点は作業前にユーザーに確認する。推測で進めない。

---

## 作業前の確認フロー

文書の**新規作成・編集**を行う前に、必ず以下の手順を踏む:

1. **現状把握**: 作業指示に基づき、関連する既存ファイルとドキュメント構造を確認する。
2. **作業計画**: 何をどのように作成・変更するかを整理し、ユーザーに提示する。
3. **承認を得る**: 方針に問題がないかユーザーの確認を取ってから作業を開始する。
4. **不足情報の確認**: 作業を進めるため、またはコンテンツの品質を高めるために追加情報が必要な場合は、作業開始前にまとめて質問する。

## 技術スタック・バージョン方針

### Unity
- **Unity 6 以降**を対象とする。
- Unity 2022 以前の API（例: `FindObjectOfType` の非推奨版）は使用しない。
- Unity 6 で追加・変更された API がある場合はそちらを優先する。

### C#
- **C# 12 以降の構文を積極的に使用する**（Unity 6 は .NET Standard 2.1 / C# 9 以降対応。Unity 6.1 以降は C# 12 対応）。
- ただし、初心者向けコンテンツでは新構文を使う際に「この書き方は C# X から使えます」と注記する。
- 以下の古い書き方は避け、モダンな構文を優先する:

  | 避けるべき書き方 | 優先する書き方 |
  |---|---|
  | `new List<int>()` | `new List<int>()` ※型推論が使える場面では `var` を使う |
  | 明示的な型宣言が冗長な場面 | `var` または型推論を活用 |
  | `string.Format(...)` | 文字列補間 `$"..."` |
  | `null` チェックの冗長な記述 | null 合体演算子 `??`, null 条件演算子 `?.` |

### Jekyll
- テーマ: **minima**（`docs/_config.yml` 参照）
- `baseurl: "/unity-csharp-learning"` が設定されているため、内部リンクは以下の形式を使う:
  - Markdown リンク: `[テキスト](/unity-csharp-learning/パス/)`

### ファイル・ページ命名規則

- **ページフォルダー**: `kebab-case/`（番号プレフィックスなし。例: `variables/`, `conditionals/`）
- **ページ本体**: フォルダー内の `index.md`
- **ページ内アセット**: 同じフォルダー内に配置（例: `variables/type-chart.png`）
- **ページの順序**: 番号はフォルダー名に付けず、親セクションの `index.md` のトピック一覧で管理する

## コンテンツ作成ルール

### 新規ページの作成手順

1. `docs/_template.md` をコピーしてファイルを作成する。
2. frontmatter の `title`, `permalink` を設定する。
3. 対応するセクションの `index.md` のトピック一覧に追加する。

### frontmatter 必須項目

```yaml
---
layout: page
title: ページタイトル
permalink: /セクション名/ファイル名/
---
```

### ページ構成

各ページは以下のセクションを含む（**太字**が必須、それ以外は任意）:

1. **学習目標** — 読み終えた後にできるようになること（箇条書き）
2. **前提知識** — 必要な前のページへのリンク
3. **本文** — 概念説明とコード例（シンプル → 実践的の順）
4. よくあるミス — 初心者がはまりやすいパターンと正しい書き方（`// ❌ NG:` / `// ✅ OK:` 形式）
5. ワンポイントアドバイス — 本題の流れで紹介する関連機能・周辺知識
6. **まとめ** — 学んだことの箇条書き
7. **理解度チェック** — 確認問題と `<details>` で隠した解答
8. **次のステップ** — 次のページへのリンク

### 構文・API の初出紹介ルール

ページ内で**初めて登場する構文や API** は、使用例の直前に以下のブロックを挿入する。

**API の場合**（メソッド・プロパティ・コンストラクターなど）:

1. **太字の名前** と一行の説明（日本語）
2. 公式ドキュメントへのリンクのプレースホルダー `<!-- [公式ドキュメント]() -->` を残す（**リンク URL はユーザーが記入する。エージェントは URL を書かない**）
3. `**書式：クラス名.メンバー名 種別**` ラベルに続いて `csharp` コードブロックでシグネチャを表示
4. パラメータ表（名前・型・説明）

**C# 文法の場合**（演算式・宣言・ステートメントなど）:

1. `**書式：文法の名前**` ラベルに続いて**プレーンコードブロック**（言語指定なし）で書式パターンを表示
   - パターン内の要素名は日本語の仮変数でよい（例: `型`・`変数名`・`式`）
   - C# 仕様書への厳密な準拠は不要。初心者にわかりやすい表現を優先する
2. 要素の説明表（名前・説明）

例:

**書式：変数宣言**
```
型 変数名;
```

| 要素 | 説明 |
|---|---|
| `型` | 変数に入れられる値の種類（`int`、`string` など）|
| `変数名` | プログラム内で変数を識別するための名前 |

同じ構文・API が同一ページ内に再登場する場合は繰り返さない。

### コード例のルール

- **必ず動作するコードを書く**。動作確認ができないコードは載せない。
- 最初の例はできる限りシンプルにし、徐々に複雑にする。
- `// ❌ NG:` と `// ✅ OK:` のパターンで誤用例と正しい例を対比させる。
- Unity API を使うサンプルは、コンテキストが明確になるよう `MonoBehaviour` を継承したクラスとして書く。

### 命名規則

- **`private` フィールドはアンダースコア始まりの `_camelCase` にする**（例: `_speed`、`_isGrounded`、`_startTime`）。
  - `[SerializeField]` を付けた `private` フィールドも同様（例: `[SerializeField] private float _speed = 5f;`）。
  - ローカル変数・パラメーターはアンダースコアなしの `camelCase`（例: `float elapsed`、`int index`）。
  - `public` フィールド・プロパティは `PascalCase`（例: `public int Score`）。

### 難易度・学習ステップ

- **一度に一つの概念**だけを導入する。複数の新しい概念を同時に使うコード例は避ける。
- 抽象的な説明の前に、必ず具体例を先に示す。
- 初心者が「なぜそう書くのか」を理解できるよう、背景や理由を添える。

## やってはいけないこと

- `docs/_config.yml` の `baseurl` / `url` を無断で変更しない。
- `_template.md` は Jekyll の仕様（`_` で始まるファイルはビルド出力に含まれない）により公開されない。ファイルを削除しない。
